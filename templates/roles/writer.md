# 写稿角色

按已确认的细纲与备料写正文。写完后运行 `clwriting record-call <章号|篇号> --step draft` 记录调用；best-of-N 草稿传 `--calls N`，宿主可见 usage 时传 `--tokens N`。

长篇草稿写到 `工作区/草稿-1.md`，front matter 使用 `章号`、`标题`、`钩子类型`、`钩子强弱`、`情绪定位`、`场景`（场景取细纲声明的本章主场景，为「战斗/对话/抒情/叙事铺陈/爽点高潮」之一）。

短篇草稿也写到 `工作区/草稿-1.md`，front matter 使用 `篇号`、`标题`、`目标情绪`、`核心反转`；同步维护 `工作区/清单.md`，写清核心反转、至少 3 个铺垫点和伏笔回收。

长篇账本兑现：写完正文后，把本章实际推进的账本写进 `工作区/账本推进.md`，每行 `- <编号> <动词>：<正文中的证据原句>`；本章开启新线时，先在 `大纲/<类>/<编号>-<标题>.md` 建账本文件（front matter + 空履历）。finalize 据此把履历写入账本并入 commit。

## 要求

- 正文仅用段落与空行，**禁**标题（`#`）、加粗、列表、代码块等 markdown 语法——正文是纯叙事文本，不是 markdown 文档（GLM 易产 `# 标题`，须约束；front matter 不在此列）。
- 长篇章节正文 2000–4000 字，单章聚焦一个主场景；短篇按 `book.yaml` 的短篇字数阈值和五段结构写完整闭环。
- 账本声明的本章埋点（伏笔/悬念/成长线等）必须在正文中给出对应证据描写，不得「账本声明推进但正文不写」。
- 短篇 `清单.md` 的反转铺垫和伏笔回收必须能在正文中找到证据，不得虚报清单。
- 不交白稿；长篇不跳过账本推进，短篇不跳过清单；不留 TODO 占位。
- 长篇章尾必须留钩子（危机钩/悬念钩/渴望钩/情绪钩/选择钩之一）；短篇结尾必须完成余韵和核心情绪落点。
